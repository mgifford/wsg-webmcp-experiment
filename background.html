Project background

Turning guideline text into something an agent can query directly.

This project explores whether the W3C Web Sustainability Guidelines can be exposed as structured, task-oriented tools for AI agents using WebMCP.

The goal is not to replace the Web Sustainability Guidelines, human review, or professional judgement. The goal is to test whether web standards can provide machine-readable interfaces that help people find, apply, test, and document sustainability guidance more effectively.

Why WebMCP matters

Most web standards are published as documents. People read them, interpret them, and decide how to apply them. That model remains essential, but it creates friction when teams need to use standards during design reviews, procurement, development, audits, and reporting.

WebMCP introduces a possible task layer for the web. A web page can expose named tools with descriptions, input schemas, and executable functions. Compatible agents can discover and call those tools directly.

What this project uses

guidelines.json

Categories, guidelines, criteria, benefits, tags, and resources.

star.json

Testing and reporting techniques that link back to WSG guidance.

These files are copied into this project through a scheduled GitHub Action so the demo can be tested from GitHub Pages without requiring a server.

Why both WSG and STAR are useful

The WSG data explains what sustainable web practice should address and why it matters. STAR adds support for how guidance can be tested, reviewed, or evidenced.

Used together, they support a stronger workflow:

Guidance
  ↓
Checklist
  ↓
Testing method
  ↓
Evidence
  ↓
Review or reporting

This is more useful than search alone. Search helps someone find text. A STAR-enhanced checklist helps someone apply the guidance in a review, procurement, development, or audit context.

From document navigation to task completion

A large standard can be difficult to use in practice. A person may need to know which category to inspect, which tags matter, which criteria apply, and which resources are relevant.

  • Find relevant sustainability guidance for a topic.
  • Generate draft review checklist items.
  • Find STAR techniques related to a WSG guideline.
  • Check whether STAR references still align with WSG anchors.
  • Generate draft procurement requirements.
  • Generate draft audit questions.
  • Generate a draft conformance claim for human review.

What WebMCP adds

The visible demo works in ordinary browsers because it uses normal JavaScript. Firefox can load the JSON files, run searches, and display results.

WebMCP adds a separate machine-facing interface. In a browser where WebMCP is available, the page can register structured tools that compatible AI agents or browser tooling can discover and invoke. 

Normal browser interface:
A person clicks buttons and reads results.

WebMCP interface:
An agent discovers tools, reads input schemas, and calls the page's functions directly.

Example scenario

A procurement officer wants sustainability language for an accessibility-related digital service contract.

With this experiment, an AI-assisted workflow could call tools such as: 

wsg.search({ "query": "accessibility" })

wsg.suggest_procurement_requirements({
  "tag": "Accessibility",
  "limit": 10
})

wsg.generate_review_checklist_with_tests({
  "topic": "accessibility",
  "role": "procurement",
  "limit": 10
})

The result is still a draft. It still requires human review. But it is grounded in the current WSG and STAR data rather than being generated from model memory alone.

Alignment checks

This project also checks whether STAR references still align with WSG anchors. This matters because the usefulness of STAR depends on its continued relationship to the current WSG structure.

If the WSG URL anchors change, the alignment tool will surface those differences so the dataset can be refreshed and reviewed.

Current limitations

  • It is an experimental prototype, not an official W3C deliverable.
  • It does not certify conformance, sustainability, accessibility, procurement, or legal compliance.
  • Human review is required before relying on any generated output.
  • WebMCP is experimental and browser support is limited.
  • The browser cannot directly expose all internal flag state to a web page.

Why this matters for web standards

The larger question is not only whether the Web Sustainability Guidelines can use WebMCP. The larger question is whether web standards can expose authoritative, structured task interfaces for AI-assisted systems.

This pattern could eventually apply to other standards and guidance documents: accessibility standards, ARIA roles, design tokens, privacy guidance, security guidance, or digital public goods criteria.

In short, the useful model is:

Standard
  ↓
Structured data
  ↓
Accessible tool
  ↓
Human review
  ↓
Safer decisions

This experimental project was developed with assistance from generative AI tools. Human review and validation remain essential.
